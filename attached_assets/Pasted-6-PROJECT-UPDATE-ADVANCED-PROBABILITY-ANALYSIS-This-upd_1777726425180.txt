FEATURE 1 — MONTE CARLO SIMULATION MODE

Purpose:

Run extremely large pack simulations quickly to analyze probability distribution.

This is a common technique used in game design and statistics.

---

FUNCTION NAME

Add a new function to the Engine class:

runMonteCarloSimulation(packCount)

Location:

engine/engine.js

---

FUNCTION BEHAVIOR

This function must:

1) reset simulation state
2) run pack simulations in a tight loop
3) skip UI updates during the loop
4) generate a statistical report
5) output the report to the simulation log

---

SIMULATION LOOP

Pseudo logic:

for (let i = 0; i < packCount; i++)  
    run pack generation logic

The simulation should reuse the same pack rules already implemented.

Do NOT duplicate pack logic.

---

PERFORMANCE REQUIREMENT

During Monte Carlo simulation:

Do NOT update the UI after every pack.

Instead:

run all packs first  
then update the UI once

This ensures the simulation runs very quickly.

---

REPORT GENERATION

After the simulation finishes, generate a probability report.

Example log output:

Monte Carlo Simulation Complete

Packs simulated: 10,000  
Cards generated: 50,000  

Expected Legendary Rate: 1%  
Actual Legendary Rate: 0.98%

Expected Epic Rate: 4%  
Actual Epic Rate: 4.05%

Expected Rare Rate: 25%  
Actual Rare Rate: 24.91%

Expected Common Rate: 70%  
Actual Common Rate: 70.06%

Actual rates must be calculated using:

rarityCount / totalCards

---

NEW UI BUTTON

Add a button to index.html.

Button text:

Run 10,000 Pack Test

When clicked:

engine.runMonteCarloSimulation(10000)

This button is for probability stress testing.

FEATURE 2 — PACK SLOT ANALYTICS

Purpose:

Track which pack slot produces rare hits.

Real TCG systems analyze this data to validate slot design.

---

STATE UPDATE

Extend the simulation state structure.

File:

engine/state.js

Add a new property:

slotStats

Example structure:

slotStats: {
 slot1: { epic: 0, legendary: 0 },
 slot2: { epic: 0, legendary: 0 },
 slot3: { epic: 0, legendary: 0 },
 slot4: { epic: 0, legendary: 0 },
 slot5: { epic: 0, legendary: 0 }
}

---

RULE UPDATE

When generating a card:

If rarity is Epic or Legendary:

increment the corresponding slot counter.

Example:

Slot 4 generates Epic.

Increment:

slotStats.slot4.epic++

---

LOG OUTPUT

At the end of Monte Carlo simulation, print slot statistics.

Example:

Hit Distribution by Slot

Slot 1: Epic 0 | Legendary 0  
Slot 2: Epic 0 | Legendary 0  
Slot 3: Epic 0 | Legendary 0  
Slot 4: Epic 102 | Legendary 15  
Slot 5: Epic 83 | Legendary 52

This helps verify the pack slot design.

---

VALIDATION EXPECTATIONS

After running 10,000 pack simulations:

You should observe:

Slots 1–3 produce zero Epic or Legendary cards.

Slots 4 and 5 produce all hits.

Slot 4 produces more Epic cards.

Slot 5 produces most Legendary cards.

This confirms the slot system works correctly.

---

UI REQUIREMENT

The slot analytics do NOT need a separate panel.

They should only appear in the Monte Carlo report log.

This keeps the UI simple during Phase 1.

---
